## 1. Fortran Standards

### 1.1 Module Structure

- Every procedure and derived type **must** live inside a module. No bare procedures.
- `implicit none` is written **once**, immediately after `module <name>` — never repeated inside contained procedures.
- `private` follows `implicit none` as the default visibility; public symbols are listed explicitly.

```fortran
module my_module
    use iso_fortran_env, only: real32, real64
    implicit none
    private

    public :: wp, my_sub

    integer, parameter :: sp = real32
    integer, parameter :: dp = real64
    integer, parameter :: wp = dp   ! working precision — change here only

contains

    subroutine my_sub(x, y)
        !> Input value
        real(wp), intent(in)  :: x
        !> Output value
        real(wp), intent(out) :: y
        ! local comment style
        y = x * 2.0_wp
    end subroutine my_sub

end module my_module
```

### 1.2 Working Precision (`wp`)

- All floating-point declarations use `real(wp)`; all literals carry the `_wp` suffix.
- Exception: when interfacing with a library that exposes its own kind parameter (e.g., `LAPACK_dp`), use that parameter directly and document why.

### 1.3 Dummy Arguments

- Every dummy argument must carry an explicit `intent(in | out | inout)`.
- Each dummy argument is declared on its own line, preceded by a `!>` doc-comment.

### 1.4 Case Conventions

- All Fortran keywords, identifiers, and intrinsics: **lowercase**.
- Exceptions (uppercase allowed):
  - Mathematical/physical symbols that follow field convention (`Ylm`, `Rnl`, `Gamma`).
  - Foreign-language directives (OpenMP `!$OMP`, CUDA, preprocessor macros).

### 1.5 Output — No `print *`

- **Never** use `print *`. Use `write(*, …)` with an explicit format string.
- Short one-off output: inline format string.

  ```fortran
  write(*, "(A, I0, A, ES12.5)") "Step: ", iter, " | Residual: ", resid
  ```

- Repeated or long format strings: declare as a `character(len=*), parameter` and reference by name.

  ```fortran
  character(len=*), parameter :: fmt_log = &
      "('Iter: ', I5, ' | Energy: ', F14.8, ' | Err: ', ES10.3)"
  write(*, fmt_log) i, energy, err
  ```

### 1.6 Comments and Documentation

| Marker | Where to use |
|--------|-------------|
| `!>` | Module/program/procedure **header** (purpose, algorithm, math); one line **above** each dummy argument declaration |
| `!` | All in-body comments; local variable descriptions (same line or line above) |

- All comments in **formal English** — self-explanatory but not verbose.
- Include LaTeX formulae in comments where the algorithm is non-trivial.

**Critical placement rule — dummy argument documentation belongs inside the procedure, not in the header.**

Do **not** bundle `@param`-style annotations into the header block above the procedure name. Instead:

- The `!>` header (above `subroutine`/`function`) describes only the procedure's purpose and algorithm.
- Each dummy argument is documented by a `!>` line placed **immediately above its own declaration**, inside the procedure body.

```fortran
! ✗ Wrong — @param annotations crammed into the header, declarations left bare
!> @brief Constructs the coefficient design matrix H = S * K.
!> @param[out] mat_H Output matrix [n_links, n_basis]
!> @param[in]  x_vec Current SLF estimate [n_grid]
subroutine build_matrix_H(mat_H, x_vec)
    real(wp), allocatable, intent(out) :: mat_H(:,:)
    real(wp),              intent(in)  :: x_vec(:)

! ✓ Correct — header describes purpose only; each argument documented at its declaration
!> Constructs the coefficient design matrix: $H = S K$,
!> where S is the per-cluster aggregation matrix. Built explicitly
!> because n_basis is small (~100–1000), making dense storage acceptable.
subroutine build_matrix_H(mat_H, x_vec)
    !> Output design matrix [n_links, n_basis]
    real(wp), allocatable, intent(out) :: mat_H(:,:)
    !> Current SLF estimate [n_grid]
    real(wp),              intent(in)  :: x_vec(:)
```

## 2. Python Standards

- Follow **PEP 8** strictly (line length ≤ 79 chars for code, ≤ 72 for docstrings/comments).
- Type hints on all function signatures; docstrings in NumPy style.
- Scientific stack imports in canonical aliases: `numpy as np`, `matplotlib.pyplot as plt`, `scipy` submodules imported explicitly.
- All comments in formal English; avoid `print` in library functions — use `logging`.
- Scripts are self-contained CLI tools (`if __name__ == "__main__":` guard always present).

---

## 3. Cross-Language Global Rules

1. **Language of all comments and documentation**: formal, accurate English.
2. **Data flow**: Fortran programs write to `outputs/`; Python scripts in `scripts/` read from `outputs/` and write figures to `outputs/plots/`.
3. **No magic numbers**: every physical or numerical constant is named and documented with units.
